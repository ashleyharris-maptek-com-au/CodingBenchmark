GLSL fragment shader (heatmap based on NdotV)
- Assumes a camera at the origin for the view vector, i.e., V = normalize(-worldPos).
- Uses the provided per-vertex inputs and writes the heatmap color to location 0.

#version 450 core
layout(location = 0) in vec3 worldPos;
layout(location = 1) in vec3 normal;
layout(location = 2) in vec2 uv;
layout(location = 3) in vec3 tangent;
layout(location = 4) in vec4 color;

layout(location = 0) out vec4 outColor;

// Optional uniform block (not strictly required for the heatmap calculation)
// You can populate cameraPos if you want to use an actual camera position instead of origin.
// layout(set = 0, binding = 0) uniform CameraBlock {
//   mat4 model;
//   mat4 view;
//   mat4 proj;
//   vec4 lightPos;
//   vec4 cameraPos; // xyz is the camera position
//   vec4 params;
// } cam;

void main() {
  // View vector from fragment to camera (camera at origin in this simple setup)
  vec3 V = normalize(-worldPos);

  // N is in world space (from input), V is in world space as well
  float ndotv = max(dot(normal, V), 0.0);

  float r, g, b;

  if (ndotv < 0.5) {
    // Low angle: map to (1, 2*NdotV, 0)
    r = 1.0;
    g = 2.0 * ndotv;
    b = 0.0;
  } else {
    // High angle: map to a two-band ramp
    float t = ndotv - 0.5;
    r = 1.0 - 2.0 * t;
    g = 1.0 - 2.0 * t;
    b = 2.0 * t;
  }

  outColor = vec4(r, g, b, 1.0);
}

If you want to use a true camera position from a uniform (instead of assuming origin), you can uncomment the CameraBlock and replace V calculation with:
  vec3 camPos = cam.cameraPos.xyz;
  V = normalize(camPos - worldPos);

Generating the SPIR-V hex string
1) Compile GLSL to SPIR-V (fragment shader):
   - Use a tool like glslangValidator.
   - Example:
     glslangValidator -V heatmap.frag -S frag -SCH -o heatmap.spv

2) Dump the SPIR-V binary as a hex string (little-endian bytes, matching SPIR-V layout):
   - On Linux/macOS, you can do:
     xxd -p -c 4 heatmap.spv | tr -d '\n'
   - This prints the bytes in order; since SPIR-V is little-endian, the 32-bit words will appear as 8 hex characters per word in little-endian byte order, exactly as you described.

3) If you prefer a single continuous hex string with no separators, you can remove newlines (the command above already does that with -d '\n').

Notes and assumptions
- The shader uses no complex UBO layout; it relies on per-vertex inputs provided by your pipeline and writes out a final color at location 0, as requested.
- The exact SPIR-V hex will depend on the compiler and any minor alignment/IDs produced by the assembler. The GLSL code above is a faithful translation of your heatmap logic and will generate a valid SPIR-V binary when compiled with a standard toolchain.
