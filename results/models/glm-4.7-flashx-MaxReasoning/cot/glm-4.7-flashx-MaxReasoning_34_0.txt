        *   *Clark & Wright Savings (CWS):* Good trade-off for speed vs. quality. Very standard for CVRP.
        *   *Metaheuristics (Tabu Search, Simulated Annealing, Genetic Algorithms):* Good for quality, but more complex code.

    *   `demands`: A list of demand for each node.
    *   `savings`: A priority queue (min-heap) to store potential merges (savings value, route1, route2).

5.  **Step-by-Step Implementation Plan (Clark-Wright):**
    1.  **Parse Input:** Read N, K, CAP. Read Depot coords, then Customer coords/demands.
    2.  **Distance Matrix:** Calculate Euclidean distances between all points. (Using $d^2$ or actual Euclidean, usually Euclidean is fine).
    3.  **Initial Solution:** Create K empty routes (just the depot). Since we need to minimize routes, we can try to fit customers into existing routes if possible, but standard CWS starts with single-vehicle tours (Depot -> Customer i -> Depot) for all customers and merges them.
    4.  **Savings Calculation:** For every pair of customers $i$ and $j$ where $i < j$, calculate savings $s_{ij} = d_{0i} + d_{0j} - d_{ij}$.
    5.  **Greedy Merge:** Sort savings descending. Iterate through savings. Check if route 1 and route 2 are compatible (capacity) and can be joined.
        *   Compatibility: Neither route contains the other's customers.
        *   Joining: Connect the end of route 1 to start of route 2 (or vice versa) and update costs.
    6.  **Output Formatting:** Calculate total distance, count routes, format output strings.

6.  **Refining the Algorithm (Handling Capacities):**
    *   *Initial Routes:* Instead of creating N routes, it's often better to create routes containing sets of customers if the number of routes K is small, but strictly speaking, CWS starts with $N$ tours.
    *   *Capacity Check:* When merging Route A and Route B, we need to check the total demand of the combined route. Also, we need to make sure we don't create invalid routes (like visiting a customer twice or creating a detached segment).
