    *   **Technical Constraints:**
        *   `local_size_x = 256`.
        *   Guard against $id \ge N$.
        *   Skip self-interaction ($i \neq j$).
        *   Use `gl_GlobalInvocationID.x`.
    *   **Test Configs:** Various $N$ values.

2.  **Drafting the Algorithm:**

    *   **Variables:**
        *   $N = \text{params.counts}.x$
        *   $dt = \text{params.physics}.x$
        *   $\epsilon = \text{params.physics}.y$
        *   $G = \text{params.physics}.z$

    *   **Indexing:**
        *   Body $i$ starts at index `i * 2`.
        *   Position `inData[i*2].xyz`
        *   Mass `inData[i*2].w`
        *   Velocity `inData[i*2+1].xyz`

    *   **Compute Loop:**
        *   Outer loop over $j$ (0 to $N-1$). If GLSL doesn't allow varying loops based on a uniform $N$ easily in a simple way without splitting the workgroup or using the binding directly, I need to think about optimization.
        *   *Optimization:* Pure `for (int j = 0; j < N; j++)` causes divergence for $N > 256$. While simple, it's slow.
        *   *Alternative:* Iterate over all available bodies in the buffer. Calculate acceleration for body $i$ by summing forces from body $j$. Inside that sum, skip if $j == i$.
        *   *The "Naive" Approach (easier to implement correctly first):*
            ```glsl
            for (int j = 0; j < 2048; j++) { // Max N is 2048 per test
                if (j >= N) break;
                // calc acc...
            }
            ```
        *   *The "Optimized" Approach (Interleaved):*
            Since we want to calculate the *acceleration for body $i$* given data from body $j$, and $i$ is the global ID, we can read the global buffer.
            ```glsl
            vec4 posMassJ = inData[j * 2];
            vec4 velJ = inData[j * 2 + 1];
            // check j != i
            // check j < N
            // sum acc
            ```
        *   Since $N$ is up to 2048 and local size is 256, we have roughly 8 workgroups per simulation. Doing a loop up to 2048 inside a shader running on a 256 thread workgroup is fine, though bandwidth-heavy.
        *   Let's stick to a loop over `counts.x`. Even if the inner loop diverges, GPUs are good at handling this in small batches, but for $N=2048$, it's small enough to run reasonably well on modern hardware.

    *   **Step-by-Step Logic for `gl_WorkGroupID.x * 256 + gl_LocalInvocationID.x`:**

        1.  Get `idx = gl_GlobalInvocationID.x`.
        2.  If `idx >= N`, return.
        3.  Load `pos` and `mass` from `inData[idx*2]`.
        4.  Load `vel` from `inData[idx*2+1]`.
        5.  Initialize `accel = vec3(0.0)`.
        6.  Loop $j$ from 0 to $N-1$:
            a. If $j == idx$, continue.
            b. Load `pos_j`, `mass_j` from `inData[j*2]`.
            c. Calculate vector $r = pos_j - pos$.
            d. Calculate distSq = dot(r, r) + softening^2.
            e. invDistCube = 1.0 / pow(distSq, 1.5).
            f. `accel += G * mass_j * r * invDistCube`.
        7.  Update `vel += accel * dt`.
        8.  Update `pos += vel * dt`.
        9.  Write to `outData[idx*2]` (Position, Mass) -> Note: Mass usually doesn't change, but the prompt says "Write BOTH position+mass and velocity vec4s". It implies the position field is updated. Does mass change? No. Does velocity change? Yes.
        10. Write to `outData[idx*2+1]` (Velocity, 0.0).

    *   **Refining the Integration Method:**
        The prompt says "using the leapfrog/Euler integration method". Then it specifies: "Velocity is advanced by acc * dt and position is advanced by vel * dt."
        This describes Semi-Implicit Euler (Symplectic Euler).
        1. $v_{new} = v_{old} + a(x_{old}) \cdot dt$
        2. $x_{new} = x_{old} + v_{new} \cdot dt$
        This is the standard standard for N-body to conserve energy better than standard Euler.

3.  **Writing the GLSL Code (Mental Draft):**

    ```glsl
    #version 450
    layout(local_size_x = 256) in;

    layout(set = 0, binding = 0) readonly buffer InputBodies {
        vec4 inData[];
    };
    layout(set = 0, binding = 1) buffer OutputBodies {
        vec4 outData[];
    };
    layout(set = 0, binding = 2) uniform Params {
        uvec4 counts;
        vec4 physics;
    };

    void main() {
        uint idx = gl_GlobalInvocationID.x;
        if (idx >= counts.x) return;

        // Physics params
        float dt = physics.x;
        float epsilon = physics.y;
        float G = physics.z;

        // Read own data
        vec4 posMass = inData[idx * 2];
        vec4 vel = inData[idx * 2 + 1];

        vec3 pos = posMass.xyz;
        float mass = posMass.w;

        vec3 accel = vec3(0.0);
        uint N = counts.x;

        // Accumulate forces
        for (uint j = 0; j < N; ++j) {
            if (j == idx) continue;

            vec4 otherPosMass = inData[j * 2];
            vec3 r = otherPosMass.xyz - pos;
            float r2 = dot(r, r) + epsilon * epsilon;
            // Optimization: r2 could be close to zero
            // float invDistCube = 1.0 / (pow(r2, 1.5)); 
            // Alternatively: (r / r2) * sqrt(1.0/r2)
            
            // Using pow or sqrt to be explicit/clear, GPU does this fast
            float invDistCube = 1.0 / pow(r2, 1.5);
            
            accel += G * otherPosMass.w * r * invDistCube;
        }

        // Integration (Semi-Implicit Euler)
        vel.xyz += accel * dt;
        pos += vel.xyz * dt;

        // Write back
        // outData[idx*2] = vec4(pos, mass) -- mass is constant
        outData[idx * 2] = vec4(pos, mass);
        outData[idx * 2 + 1] = vec4(vel.xyz, 0.0);
    }
    ```

5.  **Optimization Consideration (Optional but good):**
    The loop `for (uint j = 0; j < N; ++j)` is $O(N^2)$ complexity. With $N=2048$, that's $\approx 4$ million iterations per workgroup.
    Total work = $\lceil 2048 / 256 \rceil \times 4,000,000 \approx 8 \times 4,000,000 = 32,000,000$ scalar ops.
    On a modern GPU, 32M ops is nothing. The latency of reading from memory (Global Memory) is the bottleneck here, not the math.
    The loop structure is fine for this scale. No need to complicate with atomic adds or group shared memory optimizations (like binning) unless the prompt asked for high performance, but standard GPU simulation is usually done with naive $O(N^2)$ in GLSL if $N < 5000$ or using CUDA-style reduction in compute shaders.
    However, strictly adhering to "Each invocation handles one body" implies the calculation of $a_i$ happens for body $i$. Reading the full buffer for every $i$ is the definition of the $O(N^2)$ problem.
    
    One minor detail: The prompt asks for "Leapfrog/Euler". Leapfrog is $v_{1/2} = v_0 + 0.5 a_0 dt$, $x_1 = x_0 + v_{1/2} dt$, $v_1 = v_{1/2} + 0.5 a_1 dt$. This is a bit complex to implement in a single pass without storing half-steps. The "Semi-Implicit Euler" described in the prompt ("Velocity is advanced... Position is advanced...") is the standard interpretation for a simple N-body shader. I will stick to the explicit instruction in the prompt: Update $v$ by $acc * dt$, then $pos$ by $vel * dt$.

    Variables:
    `uint N = counts.x;`
    `vec4 physics` -> `physics.x`, `physics.y`, `physics.z`.
    `vec4 inData[]` -> `inData[i*2]` and `inData[i*2+1]`.

    *Buffer layout details:*
    The input struct defines `vec4 inData[]`. In GLSL, indexing `inData[i]` accesses a vec4. This matches the description perfectly.

6.  **Final Code Construction:**

    ```glsl
    #version 450

    layout(local_size_x = 256) in;

    // Input buffer layout
    layout(set = 0, binding = 0) readonly buffer InputBodies {
        vec4 inData[];
    };

    // Output buffer layout
    layout(set = 0, binding = 1) buffer OutputBodies {
        vec4 outData[];
    };

    // Parameters
    layout(set = 0, binding = 2) uniform Params {
        uvec4 counts;    // counts.x = N
        vec4 physics;    // physics.x = dt, physics.y = softening, physics.z = G
    };

    void main() {
        uint idx = gl_GlobalInvocationID.x;

        // Guard against extra workgroups
        if (idx >= counts.x) {
            return;
        }

        // Read current state
        // Position (xyz) and Mass (w)
        vec4 posMass = inData[idx * 2];
        // Velocity (xyz) and dummy (w)
        vec4 vel = inData[idx * 2 + 1];

        vec3 pos = posMass.xyz;
        float mass = posMass.w;
        vec3 velocity = vel.xyz;

        // Physics constants
        float dt = physics.x;
        float epsilon = physics.y;
        float G = physics.z;

        uint N = counts.x;
        vec3 acceleration = vec3(0.0);

        // Compute acceleration from all other bodies
        for (uint j = 0u; j < N; ++j) {
            if (j == idx) {
                continue;
            }

            // Read body j data
            vec4 otherPosMass = inData[j * 2];
            
            // Vector from i to j
            vec3 r = otherPosMass.xyz - pos;

            // Distance squared + softening
            float r2 = dot(r, r) + (epsilon * epsilon);

            // 1 / (r^2 + e^2)^(3/2)
            // r2 is always positive, avoiding division by zero
            float invDistCube = 1.0 / pow(r2, 1.5);

            // F = G * m_j * r_hat
            // a_i += G * m_j * (r_vec) * invDistCube
            acceleration += G * otherPosMass.w * r * invDistCube;
        }

        // Integration (Semi-Implicit Euler)
        // v = v + a * dt
        // x = x + v * dt
        velocity += acceleration * dt;
        pos += velocity * dt;

        // Write back to output buffers
        // outData[i*2] stores Position and Mass (Mass remains constant)
        outData[idx * 2] = vec4(pos, mass);
        
        // outData[i*2+1] stores Velocity and dummy
        outData[idx * 2 + 1] = vec4(velocity, 0.0);
    }
    ```
