Now code: We'll need to handle adjacency: 4 vertices (0-3). For each part (two parts: 0->1, 1->2), we'll compute tangents and emit quad.

We also need to calculate per-part colors: Interpolate between color[1] and color[2] for each part.

Actually, the problem says: "Color from color[1] and color[2], interpolated." For each quad, we could compute color at the midpoint: (color[1] + color[2]) / 2. But we need to set outColor for each emitted vertex. So we can compute color per part: col = (color[1] + color[2]) / 2.

Given that the vertex colors are passed as arrays in geometry shader input, we can retrieve them using OpAccessChain. Since the arrays are of length 4, we can index by vertex index.

But we also need to know which vertex we are emitting per quad.

Let's define:

- For each part (start_part, end_part), we have a quad of 4 vertices: lower-left (p0), lower-right (p1), upper-left (p2), upper-right (p3). So we need to emit them in order: p0 -> p1 -> p2 -> p3. With OutputTriangleStrip, we need to emit 4 vertices per quad. After each quad, we emit OpEndPrimitive.

Thus total vertices: 8.

Implementation steps:

1. Get start and end positions:
   - startPos = gl_in[1].gl_Position
   - endPos = gl_in[2].gl_Position

2. Get vertex colors:
   - col1 = worldPos_param[1] // but worldPos is vec3 (positions in clip space), not color.
   - Actually we have color[1] and color[2]:
   - col1 = color_param[1]
   - col2 = color_param[2]

3. Compute average color for each quad:
   - col_mid = (col1 + col2) / 2.0

4. Compute direction and length:
   - dir = endPos - startPos
   - len = sqrt(dot(dir, dir))
   - unit_dir = dir / len

5. Compute half thickness vector perpendicular to curve direction. Since we have 3D positions, we can compute a normal vector using cross product with up vector (0,1,0) or using third vertex (adjacency). But the problem says "same as smooth thick curve", where we compute adjacency tangents for smooth perpendiculars. Since we only have 4 vertices, we can compute the tangent at start = gl_in[2] - gl_in[1], at end = gl_in[3] - gl_in[2], then average.

Let's define:
   - tan_start = gl_in[2].gl_Position - gl_in[1].gl_Position
   - tan_end = gl_in[3].gl_Position - gl_in[2].gl_Position
   - unit_tan_start = normalize(tan_start)
   - unit_tan_end = normalize(tan_end)
   - avg_tan = (unit_tan_start + unit_tan_end) / 2.0

Now compute a perp vector:
   - perp = normalize(cross(avg_tan, vec3(0,1,0))) // might need fallback if avg_tan is parallel to up.

Alternatively, we can compute a generic up vector (0,1,0) and cross product.

Now thickness vector = perp * thickness * 0.5. Actually thickness is 0.02, half thickness is 0.01.

Thus:

   - half = thickness * 0.5 (0.01)
   - offset = perp * half

So we have four points:

   - p0 = startPos + offset
   - p1 = endPos + offset
   - p2 = startPos - offset
   - p3 = endPos - offset

But for a quad that follows a curved line, we want to offset perpendicularly. However the curve might be not straight; we could use a more complex approach.

Simpler approach: Use the quad for each segment (line segment) as above, it's okay.

Now we need to emit the first quad (first part) and second quad (third part).

Segment split: Divide the segment into 3 equal parts. For a line segment from startPos to endPos, we can compute three points:

   - pA = startPos + dir * (len * 1/3)
   - pB = startPos + dir * (len * 2/3)
   - pC = endPos

We only draw pA -> pC, but we need to fill in the missing middle part? Actually we need to draw first part (from startPos to pA) and third part (from pB to endPos), with a gap between them.

Thus we need to emit a quad for the first part (from startPos to pA) and a quad for the third part (from pB to endPos). So we need to compute per-part direction and length.

First part:

   - dir1 = pA - startPos
   - len1 = length(dir1)
   - unit_dir1 = normalize(dir1)
   - Compute perp for this direction (maybe using the average tangent again).
   - offset1 = perp * half

Third part:

   - dir3 = endPos - pB
   - len3 = length(dir3)
   - unit_dir3 = normalize(dir3)
   - offset3 = perp * half (same perp if we use same avg_tan; but we could compute a perp per direction, but okay)

We'll compute quad points for each part:

First quad points (lower-left, lower-right, upper-left, upper-right) maybe p0 = startPos + offset, p1 = pA + offset, p2 = startPos - offset, p3 = pA - offset. But careful: The quad is a triangle strip: p0 -> p1 -> p2 -> p3. Usually for a quad that follows the line direction, you offset inwards/outwards.

Alternatively, we could define the quad as:

- lower-left = startPos + offset
- lower-right = pA + offset
- upper-left = startPos - offset
- upper-right = pA - offset

Then we emit these in order for a triangle strip.

Third quad points:

- lower-left = pB + offset
- lower-right = endPos + offset
- upper-left = pB - offset
- upper-right = endPos - offset

Then we emit them.

Now we need to compute per-part color: For each part, we could compute average of col1 and col2, but maybe we want to interpolate between them for the parts. Since col1 and col2 are per-vertex colors, we can compute a linear interpolation of color based on position along the segment.

Let's compute:

- color_start = col1
- color_end = col2
- For each part, we can compute color at midpoint: For first part, color_mid = (color_start + (color_start + (color_end - color_start) * (1/3))) / 2? Actually the midpoint of the first part is at (startPos + pA)/2, so the color at that point can be linearly interpolated: col_start + (col_end - col_start) * (0.5/1) = col_start + (col_end - col_start) * 0.5 = (col_start + col_end) * 0.5. That's same as average of col1 and col2. For third part, the midpoint is at (pB + endPos)/2, which also yields (col_start + col_end) * 0.5 if using the same interpolation. So it's fine.

Thus we can compute col = (col1 + col2) * 0.5 for both parts.

Now we need to store outColor for each emitted vertex. In geometry shader, we can store to outColor variable via OpStore, which writes to a pointer. Since outColor is Output, we need to store via OpStore. However, in SPIR-V geometry shader, the per-vertex output is stored via gl_out. But for flat outputs like outColor, we need to assign to a location-decorated Output variable.

Thus we need to store to outColor using OpStore at the current vertex. Since the geometry shader is per-invocation and we have multiple emitted vertices, we need to set outColor for each emitted vertex before calling OpEmitVertex.

Thus we need to assign:

- %ptr_out_v3_outColor = OpTypePointer Output %v3float_outColor
- %outColor = OpVariable %ptr_out_v3_outColor Output

Then store:

- %outColor_val = OpLoad %v3float_outColor %outColor (or we could load current value? Actually we need to assign new value each time. So we need to store to the variable: we can compute col_mid as constant, then OpStore %outColor %col_mid. But this writes to the variable, which persists across vertices. Actually the geometry shader output variable for flat data is stored per-invocation, not per-vertex, so it's fine.

But we also need to store gl_Position to gl_out: 

- %gl_out_position = OpAccessChain %ptr_out_pv_position %gl_out %int_0

But gl_out is an array? Actually gl_out is an array of gl_PerVertex blocks. In geometry shader, gl_out is an array (size equals OutputVertices). So we need to index gl_out by vertex index. Typically geometry shaders have a gl_out array of gl_PerVertex blocks, and they store to the current vertex by using gl_out[vertexIndex] (which is a variable). In SPIR-V assembly, gl_out is an Output array of gl_PerVertex blocks. So we need to access gl_out via OpAccessChain with indices.

Thus gl_out is a pointer to an array of gl_PerVertex blocks. Actually gl_out is declared as:

- %arr_pv_out = OpTypeArray %gl_PerVertex_out %uint_8
- %ptr_out_arr_pv = OpTypePointer Output %arr_pv_out
- %gl_out = OpVariable %ptr_out_arr_pv Output

Thus to store to the current vertex (say 0-indexed), we do:

- %current_out = OpAccessChain %ptr_out_pv %gl_out %int_0

But wait: The OpAccessChain to gl_out gives us a pointer to the first element. But we need to store to the current vertex index. In geometry shader, we usually maintain a vertex counter: we increment each time we emit a vertex, and then store to gl_out[vertexCounter]. But in SPIR-V geometry shader, we can use OpAccessChain with index %int_i where i is the vertex index.

Thus we need to maintain a variable i = OpIAdd %uint i %uint_0 %uint_1.

We start i = 0 before loop.

We then compute gl_out_position = OpAccessChain %ptr_out_pv_position %gl_out %i %int_0, store gl_Position to it.

Then we compute outColor_address = OpAccessChain %ptr_out_v3_outColor %outColor (or just store to outColor variable directly). Actually outColor is a flat output variable; we can store to it using OpStore after computing its value. But for each emitted vertex, we need to store the same color to it.

Thus we can do:

- %col_mid = OpFAdd %v3float_outColor (col1 + col2) // but we need to add each component: col1 + col2 -> we need to compute vector addition.

col1 = OpLoad %v3float_outColor %worldPos? Wait worldPos is an array of vec3, not color. The color arrays are separate. So we have:

- %color_param = OpFunctionParameter %ptr_in_arr_v3_color

We can load col1 and col2:

- %col1_ptr = OpAccessChain %ptr_in_v3 %color_param %int_1
- %col1 = OpLoad %v3float_outColor %col1_ptr
- %col2_ptr = OpAccessChain %ptr_in_v3 %color_param %int_2
- %col2 = OpLoad %v3float_outColor %col2_ptr

Now compute col_mid = (col1 + col2) / 2.0.

- %col_sum = OpFAdd %v3float_outColor %col1 %col2
- %col_half = OpFMul %v3float_outColor %col_sum %const_float_0_5

Now for each vertex emit, we do:

- OpStore %outColor %col_half

But careful: outColor is a Output variable that is not per-vertex; but it's okay: each emitted vertex sees the same value (flat output). The spec says that for outputs that are not per-vertex, you set them once.

Now we need to compute gl_out_position for each vertex. However, we need to compute gl_Position for each of the 4 vertices in a quad, which might be same or different. The geometry shader expects gl_Position stored to gl_out[vertexIndex]. So we need to compute gl_Position for each vertex.

Now note: gl_Position is a vec4, and gl_in and gl_out are gl_PerVertex blocks that contain gl_Position.

Now compute unit vectors:

- %len_tan_start = OpFAdd (but we can compute length using built-in function OpDot, then sqrt via OpExtInst) or use built-in OpSqrt.

But for simplicity, we can use vector normalization: In SPIR-V, we can use OpFOrdEqual, etc. But we need to compute normalized tangent vector. We can compute length via:

We can compute length_inv = OpFDiv %float %const_float_1_0 %tan_start_len_sqrt.

Then unit_tan_start = OpFMul %v4float %tan_start_vec %length_inv.

Similarly for tan_end.

Alternatively, we can use OpVectorTimesScalar.

Now compute avg_tan = (unit_tan_start + unit_tan_end) * 0.5.

Compute length of avg_tan, then normalize:

Now compute a perpendicular vector. Since we have 3D positions, we can cross with up vector (0,1,0). But gl_Position is vec4 with w possibly 1.0 (clip space coordinates). Typically we treat the position as homogeneous coordinate; we can drop the w coordinate for cross product if needed. Or we can treat gl_Position as a 3D vector ignoring w. In SPIR-V geometry shader, gl_Position is vec4, but we can treat it as 3D (x,y,z). But cross product with a 4D vector may have issues. We can compute perp in 3D by dropping w:

- %perp = OpVectorTimesScalar %v4float (some cross product) ... but we can use OpCross that works on 3D vectors only.

Now we can compute offset vector: unit_cross * half_thickness (0.01). But offset needs to be a vec4: we can create a vec4 with w=0.

- %offset_vec4 = OpVectorTimesScalar %v4float %unit_cross %const_float_0_01

Now we need to add offset to start and end positions to compute the quad points.

We'll compute p0 = startPos + offset_vec4, p1 = pA + offset_vec4, p2 = startPos - offset_vec4, p3 = pA - offset_vec4.

But note that pA and pB are 3D positions? Actually they are points in clip space (vec4). So we can compute pA and pB as vec4.

Compute dir = endPos - startPos.

Compute length = sqrt(dot(dir, dir)). Compute dir_normalized = dir / length.

Compute pA = startPos + dir_normalized * (length * 1/3) = startPos + dir * (1/3). Actually length * (1/3) = dir * (1/3) because dir is not normalized; we want to multiply dir by factor 1/3. So pA = startPos + dir * (1/3). That's easier.

Compute pB = startPos + dir * (2/3).

Thus:

- %factor1 = OpConstantComposite %v4float %const_float_1_3 %const_float_1_3 %const_float_1_3 %const_float_0
