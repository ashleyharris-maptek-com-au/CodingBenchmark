The key calculations:

- Compute rho = sum(f[k])
- Compute ux = sum(f[k] * cx[k]) / rho
- Compute uy = sum(f[k] * cy[k]) / rho
- For equilibrium: cu = cx[k]*ux + cy[k]*uy
- feq = w[k] * rho * (1 + 3*cu + 4.5*cu*cu - 1.5*(ux*ux + uy*uy))

For streaming, I need to write f_out to binding 1 at position (y * W + x) * 9 + k (for the current cell) AND read from binding 0 at position ((y+cy[k]) * W + (x+cx[k])) * 9 + k (for periodic boundaries).

Actually, looking more carefully at the algorithm description:
"Then stream each f_out to the neighbor cell (x+cx[k], y+cy[k]) with periodic wraparound in both axes. Write to the output buffer at the corresponding neighbor index for k."

This suggests:
- For each k, we write f_out[k] to position ((y+cy[k]) * W + (x+cx[k])) * 9 + k

But wait, there's a problem with this approach. If we're computing f_out for the cell at (x, y), and we write to neighbors, then what values do we read from when computing f_out for those neighbors? We would need to read from the previous state.

Actually, I think the intended algorithm is:
1. For each cell (x, y), compute its equilibrium based on macroscopic moments
2. Apply collision: f_out[k] = f[k] + omega * (feq[k] - f[k]) for k=0..8
3. Stream: write f_out[k] to neighbor cell (x+cx[k], y+cy[k]) in output buffer

This would work because:
- We read f[k] from input buffer for the current cell
- We compute f_out based on local f
- We write f_out to neighbor cell in output buffer

So the output buffer at position (x, y) gets overwritten by reading from neighbors. This is a valid approach for a single-step compute shader.
