Let me analyze this Job-Shop Scheduling problem carefully.

## Problem Understanding

1. **Input**: J jobs, M machines. Each job has T tasks, each task has a (machine, duration) pair.
2. **Constraints**:
   - Tasks in a job must execute in order (precedence constraint)
   - A machine can only run one task at a time (resource constraint)
   - Tasks cannot be interrupted

3. **Output**: For each job, output start_time and machine for each task.

## Solution Approaches

For different scales, I need different strategies:

### Trivial Scale (2-3 jobs, 2-3 machines)
- Simple greedy or even brute force could work
- Priority-based scheduling

### Medium Scale
- List scheduling with priorities
- Critical path-based heuristics

### Large Scale (1000+ jobs, 50+ machines, 50k+ tasks)
- Must use efficient heuristics
- Priority queue-based scheduling
- Disjunctive graph approach might be too expensive
- Simple greedy with good priority rules

## Key Algorithms

### 1. Priority-based Greedy Scheduling
- Maintain ready tasks (all predecessors completed)
- Schedule based on priority (e.g., longest remaining processing time, critical path)
- Use priority queues for efficient machine selection

### 2. List Scheduling
- Create a list of tasks ordered by some priority
- Schedule tasks in order, placing each at the earliest feasible time

### 3. Shifting Bottleneck (for medium problems)
- Iteratively optimize each bottleneck machine

For extreme scale, I'll use:
- Priority queue for ready tasks
- Track machine availability times
- Simple but effective priority rules

struct Job {
    vector<Task> tasks;

## Algorithm for Large Scale

```
1. Initialize:
   - ready_tasks = all first tasks of each job
   - machine_available_time[M] = 0
   
2. While ready_tasks not empty:
   - Select task with highest priority
   - Find earliest start time = max(job_end_time, machine_available_time[machine])
   - Schedule task
   - Update machine_available_time
   - Update job's current task pointer
   - Add next task of job to ready_tasks if exists
```

## Priority Rules

Good priority rules for Job-Shop:
- **Longest Remaining Processing Time (LRPT)**: Sum of remaining durations in job
- **Shortest Processing Time (SPT)**: Minimize average completion time
- **Most Work Remaining (MWKR)**: Similar to LRPT
- **Critical Path**: Based on longest path in disjunctive graph

For large scale, I'll use a combination:
- Primary: Remaining work in job
- Secondary: Duration of current task

#include <bits/stdc++.h>
using namespace std;

int main() {
    ios_base::sync_with_stdio(false);
    cin.tie(nullptr);
    
    int J, M;
    cin >> J >> M;
    
    struct Task {
        int machine;
        int duration;
        long long start_time = 0;
    };
    
    vector<vector<Task>> jobs(J);
    vector<long long> remaining_work(J, 0);
    
    for (int j = 0; j < J; j++) {
        int T;
        cin >> T;
        jobs[j].resize(T);
        for (int t = 0; t < T; t++) {
            cin >> jobs[j][t].machine >> jobs[j][t].duration;
            remaining_work[j] += jobs[j][t].duration;
        }
    }
    
    vector<long long> job_end(J, 0);
    vector<long long> machine_end(M, 0);
    vector<int> next_task(J, 0);
    
    // Priority queue: (priority, job_id)
    // Higher priority = schedule first
    auto cmp = [&](int j1, int j2) {
        return remaining_work[j1] < remaining_work[j2]; // max heap
    };
    
    set<int, decltype(cmp)> ready_jobs(cmp);
    for (int j = 0; j < J; j++) {
        if (!jobs[j].empty()) {
            ready_jobs.insert(j);
        }
    }
    
    while (!ready_jobs.empty()) {
        // Find best job to schedule
        // Consider: priority + earliest start time
        
        int best_job = -1;
        long long best_start = LLONG_MAX;
        
        for (int j : ready_jobs) {
            int t = next_task[j];
            long long start = max(job_end[j], machine_end[jobs[j][t].machine]);
            if (best_job == -1 || start < best_start || 
                (start == best_start && remaining_work[j] > remaining_work[best_job])) {
                best_job = j;
                best_start = start;
            }
        }
        
        // Schedule best job's next task
        int j = best_job;
        int t = next_task[j];
        Task& task = jobs[j][t];
        
        long long start = max(job_end[j], machine_end[task.machine]);
        task.start_time = start;
        
        job_end[j] = start + task.duration;
        machine_end[task.machine] = start + task.duration;
        remaining_work[j] -= task.duration;
        next_task[j]++;
        
        ready_jobs.erase(j);
        if (next_task[j] < (int)jobs[j].size()) {
            ready_jobs.insert(j);
        }
    }
    
    // Output
    for (int j = 0; j < J; j++) {
        for (int t = 0; t < (int)jobs[j].size(); t++) {

#include <bits/stdc++.h>
using namespace std;

int main() {
    ios_base::sync_with_stdio(false);
    cin.tie(nullptr);
    
    int J, M;
    cin >> J >> M;
    
    struct Task {
        int machine;
        int duration;
        long long start_time = 0;
    };
    
    vector<vector<Task>> jobs(J);
    vector<long long> remaining_work(J, 0);
    
    for (int j = 0; j < J; j++) {
        int T;
        cin >> T;
        jobs[j].resize(T);
        for (int t = 0; t < T; t++) {
            cin >> jobs[j][t].machine >> jobs[j][t].duration;
            remaining_work[j] += jobs[j][t].duration;
        }
    }
    
    vector<long long> job_end(J, 0);
    vector<long long> machine_end(M, 0);
    vector<int> next_task(J, 0);
    
    // Use priority queue: (remaining_work, job_id)
    // Higher remaining work = higher priority
    priority_queue<pair<long long, int>> pq;
    
    for (int j = 0; j < J; j++) {
        if (!jobs[j].empty()) {
            pq.push({remaining_work[j], j});
        }
    }
    
    while (!pq.empty()) {
        auto [_, j] = pq.top();
        pq.pop();
        
        // Check if this job still needs scheduling (could be stale entry)
        if (next_task[j] >= (int)jobs[j].size()) continue;
        
        int t = next_task[j];
        Task& task = jobs[j][t];
        
        long long start = max(job_end[j], machine_end[task.machine]);
        task.start_time = start;
        
        job_end[j] = start + task.duration;
        machine_end[task.machine] = start + task.duration;
        remaining_work[j] -= task.duration;
        next_task[j]++;
        
        if (next_task[j] < (int)jobs[j].size()) {
            pq.push({remaining_work[j], j});
        }
    }
    
    // Output
    for (int j = 0; j < J; j++) {
        for (int t = 0; t < (int)jobs[j].size(); t++) {
            if (t > 0) cout << " ";
            cout << jobs[j][t].start_time << " " << jobs[j][t].machine;
        }
        cout << "\n";
    }
    
    return 0;
}

So the total work is O(n log n) for the heap operations plus O(stale entries) for skipping. The stale entries are bounded by O(n) total, so overall O(n log n).

            remaining_work[j] += jobs[j][t].duration;
        }

2. We push the new (remaining_work[j], j) to the queue

The old entry is gone (we popped it). So there's only one entry per "version" of remaining_work for job j.

The only way to have stale entries is if we push multiple entries without popping them. But our algorithm pushes one entry per scheduling decision, right after popping the previous entry for that job.

Actually, I see the issue now. The stale entry check IS needed because of this scenario:
1. Job j has remaining_work = 100
2. Push (100, j)
3. Another job k has remaining_work = 90, push (90, k)
4. Pop (100, j), schedule task 0 for job j, remaining_work[j] = 80, push (80, j)
5. Now the queue has (90, k), (80, j)
6. Pop (90, k), schedule task for job k
7. ...

No stale entries here. Each job has at most one entry at any time (for its next task to schedule).

But what if we skip step 4's push? Then job j would have no entry in the queue. But we always push after scheduling (if there are more tasks).

OK I think I've convinced myself that stale entries shouldn't happen with this algorithm. But the check `if (next_task[j] >= (int)jobs[j].size()) continue;` is still useful to handle completed jobs.
