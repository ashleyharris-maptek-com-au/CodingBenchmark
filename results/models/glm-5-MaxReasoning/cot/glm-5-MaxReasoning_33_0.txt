The user wants a C# solution for the Quadratic Assignment Problem (QAP).
QAP is NP-hard. This means for $N \approx 2000$, finding an exact optimal solution is impossible within 5 minutes (or even years).
The solution must be heuristic-based for larger $N$.
The constraints are:
- $N$ can be small (maybe exact solvable) or up to 2000.
- Time limit: 5 minutes.
- Input: stdin, two matrices (Flow, Distance).
- Output: total cost and permutation.

**Algorithm Strategy:**

1.  **Complexity Analysis:**
    - Calculating the cost of a single permutation takes $O(N^2)$.
    - With $N=2000$, $N^2 = 4,000,000$ operations. This is very fast (milliseconds).
    - So, we can evaluate many permutations.

2.  **Algorithm Selection:**
    - For small $N$ (e.g., $N \le 10$ or $12$), exact methods like Branch and Bound or brute force might work, but since the limit is 2000, we need a robust heuristic.
    - Simulated Annealing (SA) is a standard metaheuristic for QAP. It performs very well and scales nicely.
    - Iterative Improvement (Greedy) is fast but gets stuck in local optima.
    - Genetic Algorithms are possible but often harder to tune parameters for in a generic "unknown N" scenario.

    *Decision:* Simulated Annealing (SA) is a good candidate. It's robust and easy to implement. It can start with a random solution and improve it.

    *Alternative/Improvement:* Since $N$ can be large, initializing with a greedy heuristic (like the heuristic proposed by Müller-Merbach or similar constructive heuristics) gives a better starting point than random.

3.  **Detailed Algorithm (Simulated Annealing):**
    - **Representation:** An integer array `p` where `p[i]` is the location assigned to facility `i`.
    - **Cost Function:** $\sum_{i=0}^{N-1} \sum_{j=0}^{N-1} \text{flow}[i][j] \times \text{dist}[p[i]][p[j]]$.
    - **Neighborhood:** Swap two assignments. That is, swap `p[i]` and `p[j]`. This is standard for permutation problems.
    - **Delta Cost Calculation:**
        If we swap locations assigned to facility $u$ and $v$ (i.e., swap $p[u]$ and $p[v]$), we don't need to recalculate the whole sum.
        The change in cost is local to interactions involving $u$ and $v$.
        Let $locU = p[u]$ and $locV = p[v]$.
        The terms involving $i=u, j=v$ or $i=v, j=u$ change, plus interactions with all other $k$.
        There are standard formulas for the delta of a swap in QAP.
        Calculating delta takes $O(N)$. Since a full cost calc is $O(N^2)$, delta is much faster.
        With $N=2000$, $O(N)$ is cheap.
    - **Cooling Schedule:**
        Start temp $T_{start}$, End temp $T_{end}$.
        Number of iterations.
        Since we don't know $N$ beforehand, dynamic parameter tuning is needed or parameters scaled by $N$.
        Actually, we can use a geometric cooling schedule.

8.  **Refining the Delta Calculation for Speed:**
    Standard Loop:
    ```csharp
    long delta = 0;
    int pu = p[u], pv = p[v];
    for (int k = 0; k < N; k++) {
        if (k == u || k == v) continue;
        int pk = p[k];
        // flow u-k, k-u
        delta += flow[u][k] * (dist[pv][pk] - dist[pu][pk]);
        delta += flow[k][u] * (dist[pk][pv] - dist[pk][pu]);
        // flow v-k, k-v
        delta += flow[v][k] * (dist[pu][pk] - dist[pv][pk]);
        delta += flow[k][v] * (dist[pk][pu] - dist[pk][pv]);
    }
    // u-v interaction
    delta += flow[u][v] * (dist[pv][pu] - dist[pu][pv]);
    delta += flow[v][u] * (dist[pu][pv] - dist[pv][pu]);
    ```
    This handles asymmetry.
