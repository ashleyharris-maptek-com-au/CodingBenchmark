Let's implement Online K-Means.
1. Read first `B` points into a buffer.
2. Run K-Means++ on `B` points to initialize `K` centroids.
3. For the first `B` points, compute their assignments based on the initial centroids, output them, and do an update step.
4. For the remaining points, read chunk by chunk: assign, output, update.

Wait, if I assign the first `B` points based on the initial centroids, and then continue updating, the first `B` points might be assigned to suboptimal clusters compared to the final centroids.
Is this acceptable? Yes, for "streaming algorithms" on massive data, online assignment is the standard. The problem says "minimize within-cluster variance", which implies we want the best quality. If we want perfect quality, we need a second pass.
Since we are writing to stdout, we cannot "rewind" stdout.
So if we must output the assignment for point $i$ before reading point $i+1$, we are forced into online assignment.
BUT, the output format doesn't strictly say we must output point $i$'s assignment before reading point $i+1$. It just says "Output format: [cluster_index] as binary uint32".
If we can buffer the output, we could theoretically buffer the WHOLE output (40GB for 10B points). No, that violates memory constraints.
So we MUST output as we stream, or the judge tolerates a delayed output (which is unlikely for strict streaming judges).
Assumption: We must assign and output in a single forward pass.

To maximize quality in a single pass:
- Use a learning rate $\eta = 1 / (1 + \text{updates}_c)$. This moves centroids significantly at first, then stabilizes.
- Use a large initialization buffer (e.g., 1,000,000 points, which is only 12MB of data).
- Assign points in the init buffer ONLY AFTER processing the whole stream? No, we have to output them. We can buffer 1M assignments (4MB) easily!
- Yes! We buffer the assignments for the initialization points.
- After the stream is exhausted, we have the final centroids.
- We can then re-assign the initialization points using the final centroids, and output their assignments!
- This requires only 4MB of memory for the buffered assignments, and 12MB for the buffered points.
- This gives NEAR-OPTIMAL assignments for the whole dataset, identical to a two-pass algorithm, using only ~16MB of RAM!

This is the perfect solution!

Let's trace the logic:
1. Read header.
2. Read up to `B = 1,000,000` points into `init_points` buffer.
3. Run K-Means++ on `init_points` to get `K` centroids.
4. Continue reading the rest of the stream chunk by chunk:
   a. For each point, find the nearest centroid.
   b. Write the assignment to stdout immediately.
   c. Update the centroid using the streaming update rule $\mu_c = \mu_c + \eta (x - \mu_c)$.
5. Once the stream is finished, the centroids are fully trained.
6. Now, go back to the `init_points`. For each point in `init_points`, find the nearest centroid using the FINAL centroids.
7. Write the buffered assignments to stdout.

Wait! The output order must match the input order.
If I output the assignments for the rest of the stream during step 4, they go to stdout FIRST.
Then in step 7, I output the assignments for the init points.
This means the assignments for the first 1M points will appear at the END of the output.
This violates the "one per point" sequence order.

To fix the order, I would need to buffer the ENTIRE output of the rest of the stream, which is impossible (40GB).
So I CANNOT delay the output of the init points.
If I must output point $i$'s assignment before point $i+1$, then I am forced to assign the init points using the INITIAL centroids.

We can't output them, then change our mind.

Yes! We can train the centroids heavily on the init buffer before we start outputting.
1. Read 1M points.
2. K-Means++ on 1M points.
3. Do 10 iterations of K-Means on the 1M points IN MEMORY (assign, update, don't output).
4. Now centroids are well-trained.
5. Process the 1M points one last time: assign, OUTPUT, update.
6. Process the rest of the stream: assign, OUTPUT, update.

This is brilliant! It gives the centroids a huge head start, so the online assignments for the first 1M points are based on very good centroids. The quality will be excellent.

Let's refine this plan.
Init Buffer Size: Let's use `B = 2_000_000` points. That's 24MB of points, easily fits in memory.
For 10B points, 2M is a tiny fraction (0.02%), but enough to establish excellent centroids.

Algorithm Steps:
1. Read `num_points` and `num_clusters`.
2. Read up to `B` points into `init_points`.
3. K-Means++ seeding on `init_points`.
4. Run standard K-Means (batch) for 10 iterations on `init_points` to fully converge centroids.
5. Final Pass on `init_points`: For each point, find nearest centroid, OUTPUT assignment, and do one streaming update.
6. Stream Pass on remaining points: For each point, find nearest centroid, OUTPUT assignment, and do streaming update.

This guarantees:
- Single pass over stdin (we only read the rest of the stream once).
- Minimal memory (only `B` points in memory, ~24MB).
- High quality (centroids are well-trained before any assignment is output).
- Correct output order.
- Extremely fast (no disk I/O, just streaming).

Let's trace the centroid updates carefully.
In step 4, we do batch K-Means:
- Compute sums and counts for each cluster.
- Divide sums by counts to get new centroids.
- Repeat 10 times.

In step 5, we start the streaming phase. We need a learning rate $\eta$.
Standard streaming k-means uses $\eta_c = 1 / \text{count}_c$.
But we already have counts from step 4!
Let `count_c` be the number of points assigned to cluster $c$ in the last batch iteration.
When we process a point in step 5 or 6, we assign it to cluster $c$, then:
`count_c += 1`
`centroid_c += (point - centroid_c) / count_c`
This is exactly equivalent to computing the running mean, which is optimal for minimizing within-cluster variance!

Wait, is it equivalent?
If we assign point $x$ to cluster $c$ with current sum $S_c$ and count $N_c$:
New sum $S_c' = S_c + x$
New count $N_c' = N_c + 1$
New mean $\mu_c' = S_c' / N_c' = (N_c \mu_c + x) / (N_c + 1) = \mu_c + (x - \mu_c) / (N_c + 1)$
Yes! The update rule is:
`centroid_c += (point - centroid_c) / (count_c + 1)`
`count_c += 1`
This perfectly tracks the exact arithmetic mean of all points assigned to the cluster so far!
This is mathematically exact for the assignments made.

What if a cluster becomes empty during streaming?
If `count_c` reaches 0, we shouldn't divide by 0. But in streaming, we just don't update it.
To prevent empty clusters (which can happen if centroids drift apart), we should check if `count_c == 0` and maybe steal a point, but in K-means, empty clusters are usually just left alone or re-initialized. Given the scale, a few empty clusters out of 10,000 might happen. We'll just skip the update for empty clusters.

Let's write the code.

```rust
use std::io::{BufReader, BufWriter, Read, Write};

const INIT_BUFFER_POINTS: usize = 2_000_000;
const CHUNK_BYTES: usize = 4096;
const CHUNK_POINTS: usize = CHUNK_BYTES / 12; // 341 points per chunk

#[derive(Clone)]
struct Centroid {
    x: f64,
    y: f64,
    z: f64,
    count: u64,
}

impl Centroid {
    fn zero() -> Self {
        Centroid { x: 0.0, y: 0.0, z: 0.0, count: 0 }
    }
    #[inline]
    fn dist_sq(&self, px: f64, py: f64, pz: f64) -> f64 {
        let dx = self.x - px;
        let dy = self.y - py;
        let dz = self.z - pz;
        dx * dx + dy * dy + dz * dz
    }
}

struct Random {
    state: u64,
}

impl Random {
    fn new(seed: u64) -> Self {
        Random { state: if seed == 0 { 1 } else { seed } }
    }
    fn next(&mut self) -> u64 {
        self.state ^= self.state << 13;
        self.state ^= self.state >> 7;
        self.state ^= self.state << 17;
        self.state
    }
    fn next_f64(&mut self) -> f64 {
        (self.next() as f64) / (u64::MAX as f64)
    }
}

#[inline]
fn read_point(buf: &[u8], i: usize) -> (f64, f64, f64) {
    let off = i * 12;
    let px = f32::from_le_bytes([buf[off], buf[off+1], buf[off+2], buf[off+3]]) as f64;
    let py = f32::from_le_bytes([buf[off+4], buf[off+5], buf[off+6], buf[off+7]]) as f64;
    let pz = f32::from_le_bytes([buf[off+8], buf[off+9], buf[off+10], buf[off+11]]) as f64;
    (px, py, pz)
}

#[inline]
fn find_nearest(px: f64, py: f64, pz: f64, centroids: &[Centroid]) -> usize {
    let mut best_idx = 0;
    let mut best_dist = centroids[0].dist_sq(px, py, pz);
    for i in 1..centroids.len() {
        let d = centroids[i].dist_sq(px, py, pz);
        if d < best_dist {
            best_dist = d;
            best_idx = i;
        }
    }
    best_idx
}

fn main() {
    let stdin = std::io::stdin();
    let mut reader = BufReader::with_capacity(1 << 20, stdin.lock()); // 1MB read buffer
    let stdout = std::io::stdout();
    let mut writer = BufWriter::with_capacity(1 << 20, stdout.lock()); // 1MB write buffer

    let mut header = String::new();
    reader.read_until(b'\n', &mut header).unwrap();
    let mut parts = header.trim().split_whitespace();
    let num_points: u64 = parts.next().unwrap().parse().unwrap();
    let num_clusters: usize = parts.next().unwrap().parse().unwrap();

    if num_clusters == 0 || num_points == 0 {
        return;
    }

    let mut rng = Random::new(num_points ^ (num_clusters as u64));
    
    // 1. Read Initial Buffer
    let mut init_points = Vec::with_capacity(INIT_BUFFER_POINTS * 3);
    let mut points_read: u64 = 0;
    let init_target = std::cmp::min(num_points, INIT_BUFFER_POINTS as u64);
    
    let mut buf = [0u8; CHUNK_BYTES];
    while points_read < init_target {
        let to_read = std::cmp::min(CHUNK_BYTES, ((init_target - points_read) * 12) as usize);
        if reader.read_exact(&mut buf[..to_read]).is_err() { break; }
        let chunk_pts = (to_read / 12) as u64;
        for i in 0..chunk_pts as usize {
            let (px, py, pz) = read_point(&buf, i);
            init_points.push(px);
            init_points.push(py);
            init_points.push(pz);
        }
        points_read += chunk_pts;
    }
    let num_init = (init_points.len() / 3) as u64;

    // 2. K-Means++ Seeding
    let mut centroids = Vec::with_capacity(num_clusters);
    let first_idx = (rng.next() as usize) % (num_init as usize);
    centroids.push(Centroid {
        x: init_points[first_idx * 3],
        y: init_points[first_idx * 3 + 1],
        z: init_points[first_idx * 3 + 2],
        count: 1,
    });

    // For efficiency, compute dists in chunks
    let mut dists = vec![f64::MAX; num_init as usize];
    for c in &centroids {
        for i in 0..num_init as usize {
            let d = c.dist_sq(init_points[i*3], init_points[i*3+1], init_points[i*3+2]);
            if d < dists[i] { dists[i] = d; }
        }
    }

    while centroids.len() < num_clusters {
        let total_d: f64 = dists.iter().sum();
        if total_d == 0.0 {
            let last = centroids.last().unwrap();
            centroids.push(Centroid {
                x: last.x + rng.next_f64() * 0.001 - 0.0005,
                y: last.y + rng.next_f64() * 0.001 - 0.0005,
                z: last.z + rng.next_f64() * 0.001 - 0.0005,
                count: 1,
            });
        } else {
            let threshold = rng.next_f64() * total_d;
            let mut cum_d = 0.0;
            let mut chosen_idx = (num_init - 1) as usize;
            for (i, &d) in dists.iter().enumerate() {
                cum_d += d;
                if cum_d >= threshold { chosen_idx = i; break; }
            }
            centroids.push(Centroid {
                x: init_points[chosen_idx * 3],
                y: init_points[chosen_idx * 3 + 1],
                z: init_points[chosen_idx * 3 + 2],
                count: 1,
            });
        }
        let c = centroids.last().unwrap();
        for i in 0..num_init as usize {
            let d = c.dist_sq(init_points[i*3], init_points[i*3+1], init_points[i*3+2]);
            if d < dists[i] { dists[i] = d; }
        }
    }

    // 3. Batch K-Means on Init Buffer (10 iterations)
    // This fully stabilizes centroids before we output anything
    let mut assignments = vec![0usize; num_init as usize];
    for _ in 0..10 {
        // Compute sums and counts
        let mut sums = vec![(0.0f64, 0.0f64, 0.0f64); num_clusters];
        let mut counts = vec![0u64; num_clusters];
        
        for i in 0..num_init as usize {
            let idx = find_nearest(init_points[i*3], init_points[i*3+1], init_points[i*3+2], &centroids);
            assignments[i] = idx;
            sums[idx].0 += init_points[i*3];
            sums[idx].1 += init_points[i*3+1];
            sums[idx].2 += init_points[i*3+2];
            counts[idx] += 1;
        }
        
        // Update centroids
        for j in 0..num_clusters {
            if counts[j] > 0 {
                centroids[j].x = sums[j].0 / counts[j] as f64;
                centroids[j].y = sums[j].1 / counts[j] as f64;
                centroids[j].z = sums[j].2 / counts[j] as f64;
                centroids[j].count = counts[j];
            } else {
                centroids[j].count = 0;
            }
        }
    }

    // 4. Output Init Buffer Assignments & Streaming Update
    // We use the exact running mean update rule to seamlessly transition from batch to streaming
    for i in 0..num_init as usize {
        let idx = find_nearest(init_points[i*3], init_points[i*3+1], init_points[i*3+2], &centroids);
        let idx_u32 = idx as u32;
        writer.write_all(&idx_u32.to_le_bytes()).unwrap();
        
        // Exact streaming mean update
        let c = &mut centroids[idx];
        c.count += 1;
        c.x += (init_points[i*3] - c.x) / c.count as f64;
        c.y += (init_points[i*3+1] - c.y) / c.count as f64;
        c.z += (init_points[i*3+2] - c.z) / c.count as f64;
    }

    // Free init buffer memory
    drop(init_points);
    drop(dists);
    drop(assignments);

    // 5. Stream Remaining Points
    while points_read < num_points {
        let to_read = std::cmp::min(CHUNK_BYTES, ((num_points - points_read) * 12) as usize);
        if reader.read_exact(&mut buf[..to_read]).is_err() {
            break;
        }
        let chunk_pts = (to_read / 12) as u64;
        
        for i in 0..chunk_pts as usize {
            let (px, py, pz) = read_point(&buf, i);
            let idx = find_nearest(px, py, pz, &centroids);
            let idx_u32 = idx as u32;
            writer.write_all(&idx_u32.to_le_bytes()).unwrap();
            
            let c = &mut centroids[idx];
            c.count += 1;
            c.x += (px - c.x) / c.count as f64;
            c.y += (py - c.y) / c.count as f64;
            c.z += (pz - c.z) / c.count as f64;
        }
        points_read += chunk_pts;
    }

    writer.flush().unwrap();
}
```

7. Is the running mean update mathematically identical to batch update?
Let's trace:
Suppose we have points $p_1, p_2, p_3$.
Batch: $\mu = (p_1 + p_2 + p_3) / 3$.
Streaming:
$\mu_1 = p_1$
$\mu_2 = \mu_1 + (p_2 - \mu_1) / 2 = p_1 + (p_2 - p_1) / 2 = (p_1 + p_2) / 2$
$\mu_3 = \mu_2 + (p_3 - \mu_2) / 3 = (p_1 + p_2) / 2 + (p_3 - (p_1+p_2)/2) / 3 = (3(p_1+p_2) + 2p_3 - (p_1+p_2)) / 6 = (2(p_1+p_2) + 2p_3) / 6 = (p_1+p_2+p_3)/3$.
Yes, it is EXACTLY identical, provided the assignments don't change!
In K-Means, assignments DO change, so it's an approximation, but it's the optimal approximation for single-pass online K-means.

How to do fast approximate nearest neighbor in 3D?
We can project the 3D points onto a 1D space-filling curve (Z-order curve / Morton code)!
Sort centroids by Morton code.
For a query point, compute its Morton code, do a binary search in the sorted centroids to find the closest Morton code, and then check a small window of $\pm 50$ centroids around it.
This is incredibly easy to implement, cache-friendly, and very fast.
For 10,000 centroids, checking $\pm 50$ means 100 distance checks.
$10^{10}$ points $\times 100$ checks $= 10^{12}$ ops.
With SIMD and cache locality, this could run in ~10-20 seconds!
This is a HUGE improvement and highly likely to be fast enough.

Let's refine the Morton code idea.
A 3D point $(x, y, z)$.
We interleave the bits of $x, y, z$ to get a 64-bit or 128-bit integer.
Wait, we need to quantize the coordinates.
Since we don't know the bounds of the data upfront, we can just use the top 21 bits of the `f32` representation (or `f64`) of the coordinates.
Actually, `f32` has 1 sign bit, 8 exponent bits, 23 mantissa bits.
If we just treat the `f32` as an integer (after flipping the sign bit so negative numbers are smaller than positive), the integer order is the same as the float order!
This is a brilliant trick!
```rust
fn float_to_sortable_int(f: f32) -> u32 {
    let mut i = f.to_bits();
    if (i & 0x80000000) != 0 {
        // Negative: flip all bits
        i ^= 0xFFFFFFFF;
    } else {
        // Positive: flip sign bit
        i ^= 0x80000000;
    }
    i
}
```
