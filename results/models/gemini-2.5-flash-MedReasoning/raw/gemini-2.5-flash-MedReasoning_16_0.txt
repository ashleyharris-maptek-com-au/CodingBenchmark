The solution to the Job-Shop Scheduling problem is provided in the `main.cpp` file.

**Explanation of the Approach:**

This solution implements a greedy scheduling algorithm for the Job-Shop Scheduling problem. The core idea is to always schedule the task that can start earliest, considering both machine availability and job sequence constraints.

**Data Structures:**
- `Task`: Represents an individual task with `machine_id`, `duration`, `start_time`, `job_id`, and `task_id_in_job` (for sequence within a job).
- `Job`: Contains a vector of `Task` pointers for a given job.
- `ProblemInstance`: Holds all input data (number of jobs, machines) and parsed `Job` objects.

**Input Parsing (`read_input` function):**
- Reads the number of jobs and machines.
- Iterates through each job, reading its tasks (machine and duration) and creating `Task` objects dynamically.

**Greedy Scheduling (`solve_greedy` function):**
1.  **State Tracking:**
    - `machine_finish_time`: A vector tracking when each machine becomes free.
    - `job_next_available_time`: A vector tracking when the next task for a particular job can start (i.e., after its previous task completes).
2.  **Ready Task Priority Queue:**
    - A `std::priority_queue` (`ready_tasks`) stores `ReadyTask` objects. A `ReadyTask` encapsulates a `Task*` and its `earliest_start` time.
    - The priority queue is ordered to always return the task with the smallest `earliest_start` time. Tie-breaking is done deterministically by `job_id` then `task_id_in_job`.
3.  **Scheduling Loop:**
    - Initially, the first task of each job (that has tasks) is added to `ready_tasks` with an `earliest_start` of 0.
    - The loop continues as long as there are tasks to schedule and the `ready_tasks` queue is not empty.
    - In each iteration:
        - The task with the earliest possible start time is extracted from `ready_tasks`.
        - Its actual `start_time` is calculated by taking the maximum of:
            - Its `earliest_start` (from the priority queue).
            - The `machine_finish_time` of its assigned machine.
            - The `job_next_available_time` for its job.
        - The task's `start_time` is set, and its `finish_time` is calculated.
        - `machine_finish_time` and `job_next_available_time` are updated.
        - If there is a next task in the same job, it is added to `ready_tasks` with its `earliest_start` set to the current task's `finish_time`.

**Output Formatting (`print_output` function):**
- Iterates through each job and its scheduled tasks, printing the `start_time` and `machine_id` for each task as specified.

**Adaptive Logic (Commented in `main` function):**
The problem statement requires adaptive logic. While the current greedy approach is a general-purpose heuristic, for truly "trivial" problems, a more exhaustive search (e.g., Branch and Bound) could guarantee optimality. For "extreme" cases, more sophisticated metaheuristics (e.g., Simulated Annealing, Genetic Algorithms, Shifting Bottleneck) would be required to find good solutions within practical time limits. The choice of strategy could be made based on parameters like `num_jobs`, `num_machines`, or total number of tasks. This current implementation serves as a robust baseline.

**Memory Management:**
Dynamically allocated `Task` objects are deallocated at the end of the `main` function to prevent memory leaks.
