The user wants a SPIR-V assembly text for a fragment shader that implements Phong lighting.
Let's break down the requirements:
- Target: Vulkan 1.0, SPIR-V 1.0.
- Output: `outColor` at Location 0.
- Inputs:
  - `worldPos` (Location 0)
  - `normal` (Location 1)
  - `uv` (Location 2)
  - `tangent` (Location 3)
  - `color` (Location 4)
- Uniform Buffer:
  - DescriptorSet 0, Binding 0
  - `model` offset 0 (mat4)
  - `view` offset 64 (mat4)
  - `projection` offset 128 (mat4)
  - `lightPos` offset 192 (vec4)
  - `cameraPos` offset 208 (vec4)
  - `params` offset 224 (vec4)
- Lighting calculation:
  - Ambient: (0.1, 0.05, 0.05)
  - Diffuse: vertex color * max(dot(N, L), 0)
  - Specular: (1, 1, 1) * pow(max(dot(R, V), 0), 32)
    - R = reflect(-L, N)
  - Sum them up, clamp to [0,1], alpha = 1.0.

Let's write the SPIR-V assembly step by step.

1. Header and Capabilities
```spirv
; SPIR-V
; Version: 1.0
OpCapability Shader
OpMemoryModel Logical GLSL450
OpEntryPoint Fragment %main "main" %worldPos %normal %uv %tangent %color %outColor
OpExecutionMode %main OriginUpperLeft
```

Wait, the vertex colors are smoothly varying, typical value rgb = 0.5+0.5*xyz. Since we clamp to [0,1], the diffuse could be up to 1.0. The sum is ambient + diffuse + specular, clamped to [0,1].

5. Function logic
```spirv
%main = OpFunction %void None %functy
%entry = OpLabel

; Load normal
%N_vec3 = OpLoad %v3f %normal
%N = OpExtInst %v3f %glsl_std450 Normalize %N_vec3

; Load worldPos
%P = OpLoad %v3f %worldPos

; Load lightPos (xyz)
%lightPos_ptr = OpAccessChain %ptr_uni_v4f %ubo %int_3
%lightPos_v4 = OpLoad %v4f %lightPos_ptr
%L_vec3 = OpVectorShuffle %v3f %lightPos_v4 %lightPos_v4 0 1 2
%L_len = OpExtInst %f %glsl_std450 Length %L_vec3
%L = OpExtInst %v3f %glsl_std450 Normalize %L_vec3

; Load cameraPos (xyz)
%camPos_ptr = OpAccessChain %ptr_uni_v4f %ubo %int_4
%camPos_v4 = OpLoad %v4f %camPos_ptr
%V_vec3 = OpVectorShuffle %v3f %camPos_v4 %camPos_v4 0 1 2
%V = OpExtInst %v3f %glsl_std450 Normalize %V_vec3

; Wait, V should be view direction from surface to camera!
; Actually, V = normalize(cameraPos - worldPos)
%P_to_C = OpFSub %v3f %V_vec3 %P
%V = OpExtInst %v3f %glsl_std450 Normalize %P_to_C

; Wait! The spec says: "specular uses (1,1,1) with shininess 32 and R=reflect(-L,N)"
; Let's follow the exact instructions!
; R = reflect(-L, N)
%minus_L = OpFNegate %v3f %L
%R = OpExtInst %v3f %glsl_std450 Reflect %minus_L %N

; Diffuse = color * max(dot(N, L), 0)
%NDotL = OpDot %f %N %L
%NDotL_clamped = OpExtInst %f %glsl_std450 FMax %NDotL %zero
%color_v3 = OpLoad %v3f %color
%diffuse = OpVectorTimesScalar %v3f %color_v3 %NDotL_clamped

; Specular = (1,1,1) * pow(max(dot(R, V), 0), 32)
%RDotV = OpDot %f %R %V
%RDotV_clamped = OpExtInst %f %glsl_std450 FMax %RDotV %zero
%spec_pow = OpExtInst %f %glsl_std450 Pow %RDotV_clamped %shininess
%specular = OpVectorTimesScalar %v3f %specular_color %spec_pow

; Sum
%sum1 = OpFAdd %v3f %ambient %diffuse
%sum2 = OpFAdd %v3f %sum1 %specular

; Clamp to [0, 1]
%final_color = OpExtInst %v3f %glsl_std450 FClamp %sum2 %zero %one

; Output
%final_v4 = OpCompositeExtract %v4f %final_color %one
; Wait, OpCompositeExtract is not for constructing. Use OpCompositeInsert or OpCompositeConstruct.
%final_v4 = OpCompositeConstruct %v4f %final_color %one

OpStore %outColor %final_v4
OpReturn
OpFunctionEnd
```

Let's double check instructions.
`OpFSub` takes result type, operand 1, operand 2.
`OpFNegate` takes result type, operand.
`OpDot` takes result type, vector 1, vector 2.
`OpExtInst` takes result type, set, instruction, operands.
`OpVectorTimesScalar` takes result type, vector, scalar.
`OpCompositeConstruct` takes result type, constituents.

Let's write out the full SPIR-V text properly.

Constants needed:
```spirv
%int_3 = OpConstant %i32 3
%int_4 = OpConstant %i32 4
%i32 = OpTypeInt 32 1
```

Is `V` correctly calculated? "specular uses (1,1,1) with shininess 32 and R=reflect(-L,N)". I should use `cameraPos - worldPos` for `V` to make it physically correct, or just normalize `cameraPos.xyz`. Usually Phong uses `normalize(cameraPos - worldPos)`. Since the camera is at (0,0,3) and sphere is a unit sphere, `cameraPos` is not a direction vector. So `normalize(cameraPos - worldPos)` is right.

Let's re-verify the reflect instruction.
`OpExtInst %v3f %glsl_std450 Reflect %I %N`
For `Reflect(I, N)`, it returns `I - 2 * dot(N, I) * N`.
If `I` is `-L`, then `Reflect(-L, N)` = `-L - 2 * dot(N, -L) * N` = `-L + 2 * dot(N, L) * N`.
This perfectly matches `R = reflect(-L, N)`.

Let's check `FClamp`. `FClamp(x, minVal, maxVal)` requires vector types for x, minVal, maxVal.
Yes, `%zero` and `%one` are scalar, but `FClamp` supports scalar promotion in GLSL.std.450? Actually, it's safer to pass vectors for min and max.
```spirv
%zero_v3 = OpCompositeConstruct %v3f %zero %zero %zero
%one_v3 = OpCompositeConstruct %v3f %one %one %one
%final_color = OpExtInst %v3f %glsl_std450 FClamp %sum2 %zero_v3 %one_v3
```
This is much safer and explicitly correct in SPIR-V.

Let's review the GLSL equivalent to ensure no logic errors:
```glsl
#version 450

layout(location = 0) in vec3 worldPos;
layout(location = 1) in vec3 normal;
layout(location = 2) in vec2 uv;
layout(location = 3) in vec3 tangent;
layout(location = 4) in vec3 color;

layout(set = 0, binding = 0) uniform UBO {
    mat4 model;
    mat4 view;
    mat4 projection;
    vec4 lightPos;
    vec4 cameraPos;
    vec4 params;
} ubo;

layout(location = 0) out vec4 outColor;

void main() {
    vec3 N = normalize(normal);
    vec3 L = normalize(ubo.lightPos.xyz);
    vec3 V = normalize(ubo.cameraPos.xyz - worldPos);
    vec3 R = reflect(-L, N);
    
    vec3 ambient = vec3(0.1, 0.05, 0.05);
    float diff = max(dot(N, L), 0.0);
    vec3 diffuse = color * diff;
    
    float spec = pow(max(dot(R, V), 0.0), 32.0);
    vec3 specular = vec3(1.0) * spec;
    
    vec3 result = clamp(ambient + diffuse + specular, 0.0, 1.0);
    outColor = vec4(result, 1.0);
}
```
This is straightforward and matches the task.

Wait, I should check `FClamp` spec.
`FClamp` requires that `x`, `minVal`, and `maxVal` all have the same type. So passing `%zero_v3` and `%one_v3` is correct.

Let me double check the `OpCompositeConstruct` instruction for `%final_v4`.
`OpCompositeConstruct %v4f %final_color %one` takes a `v3f` and an `f`, and produces a `v4f`. This is perfectly valid in SPIR-V.
