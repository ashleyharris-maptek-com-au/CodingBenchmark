For `OpAtomicUMin`, all threads are trying to write 0. Even if there is reordering, the only possible values written are 0. Since it's an atomic operation, the hardware ensures that `atomicMin` behaves as an atomic read-modify-write. No other non-atomic memory operations depend on the exact point this happens, except that before `OpAtomicIAdd`, the value must be 0. But wait, if `OpAtomicIAdd` is reordered BEFORE `OpAtomicUMin` by the hardware (because both are Relaxed), then `OpAtomicIAdd` might add to the GARBAGE value!
Oh! That is a critical bug!
If we use Relaxed semantics, the compiler/hardware CAN reorder `OpAtomicIAdd` before `OpAtomicUMin` within the SAME thread!
To prevent reordering, we MUST use memory semantics that enforce ordering.
Specifically, `OpAtomicUMin` must have at least `Release` (0x4) semantics, and `OpAtomicIAdd` must have at least `Acquire` (0x2) semantics, or we can just use `AcquireRelease` (0x6) for both, or `SequentiallyConsistent` (0x8) for both.
Let's use `AcquireRelease` (0x6) for both to be absolutely safe.
Wait, if I use `AcquireRelease` for `OpAtomicUMin`, it ensures no memory operations after it can be reordered before it.
And `AcquireRelease` for `OpAtomicIAdd` ensures no memory operations before it can be reordered after it.
So the pair will strictly maintain order.
Wait, `OpAtomicUMin` doesn't need to acquire anything, it's just releasing the "reset to 0" state. `OpAtomicIAdd` needs to acquire the fact that it's 0. So `Release` for Min and `Acquire` for Add is theoretically sufficient.
