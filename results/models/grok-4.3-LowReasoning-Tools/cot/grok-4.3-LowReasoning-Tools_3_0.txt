The task is to write a Rust solver for Graph Layout problem, specifically Edge Crossing Minimization, by outputting x y coordinates for nodes.
The task is to write a Rust program that takes a graph from stdin and outputs node coordinates to minimize edge crossings, i.e., a graph layout.

It must handle from small to very large graphs up to 200,000 nodes within 30 seconds.

I need an efficient algorithm that scales.

For large graphs, exact 0-crossing if planar is hard, NP-hard, so heuristic.

A good scalable heuristic is to use a force-directed layout with approximations for large N.

But in Rust, implementing Barnes-Hut for repulsion would be complex for time.

Simpler: use a grid layout or spectral layout but spectral needs eigenvalues, no lib 1.92, need to implement.

No external crates, pure Rust.

So, basic:

- Read N, M

- Read edges, build adj list: Vec<Vec<usize>>

- Handle multiples/self: use HashSet or just ignore.

For layout:

- If N <= 100, use simple iterative force-directed 2D with all-pair repulsion (O(N^2) per iter, for N=100, 10k, *50 iters = 0.5M ops, fine)

- For larger N, use a different method: e.g., place on a circle, or grid.

For grid: compute w = (n as f64).sqrt() as usize, place in rows.

But for disconnected, better to place components separately.

To improve: find connected components, layout each separately, place them spaced apart.

For large, if component large, grid it.

For minimal crossings, if graph is planar, grid may cross 0 if lucky, but not always.

But problem allows reasonable, not optimal.

Also, for trees, can do layered layout.

But to keep simple, implement component-aware grid placement.
