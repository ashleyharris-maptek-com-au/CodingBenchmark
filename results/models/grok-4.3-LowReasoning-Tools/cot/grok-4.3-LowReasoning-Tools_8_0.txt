The task is to write a Rust maze solver that handles mazes from 5x5 up to 50,000x50,000 efficiently. It must read from stdin: first line H W, then H lines of the maze.
The problem is to write a Rust program that solves a maze from stdin, handling up to 50,000 x 50,000 size. That's huge, 2.5e9 cells. Each cell if stored as bool for visited, 2.5e9 bytes ~2.5GB, but with 128GB RAM, possible but tight. For path, if I use BFS, queue could be large, and to reconstruct path without parent pointers (which would take too much memory), I need another way.

For any path, not shortest, I can use DFS iterative with a stack, and keep track of the path by storing the sequence in a Vec, but if path is long, memory issue again.

But for output, the path can be up to H*W long, each line "x y" ~10 bytes, so for 1e6 path length, 10MB output, ok, but for 2.5 9, impossible.

So, probably the test cases have small paths or small mazes, or mazes where path is short.

The problem says "from trivial to ludicrous scale", but "must work efficiently across ALL scales".
